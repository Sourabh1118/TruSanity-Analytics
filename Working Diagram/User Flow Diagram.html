Two-Tiered User Architecture & Flow

Visualizing the Role-Based Access Control (RBAC) and Multi-Tenant Isolation bridging across Next.js Middleware and PostgreSQL Drizzle schemas.

1. Identity & Registration (NextAuth + Postgres)

Platform Owner (You)

SUPER_ADMIN

Hardcoded via server scripts (seed-admin.ts). Because their global users.role is explicitly elevated, they bypass tenant boundaries and own the actual SaaS product.

SaaS Customer Sign-Up

USER

When someone registers on the marketing site, a database transaction automatically assigns them standard user rights globally. They cannot see other peoples' companies.

Db Transaction: 1. Creates User entry
2. Creates new isolated 'Tenant' Workspace
3. Binds them to workspace as 'Owner'

2. The Edge Middleware Network (Global Filter)

When someone tries to load an admin page, the browser checks their NextAuth JWT Cookie mathematically in milliseconds—before the page rendering engine even wakes up.

req.auth?.user?.role === 'SUPER_ADMIN'
If TRUE Granted access to `/admin` dashboard. Views Global Server Health, MRR, Platform Settings.
If FALSE Forcibly blocked! Redirected to `/dashboard`. Only sees their own isolated tenant metrics.

3. Workspace Tenants (The Isolated Bubble)

Once a regular user is pushed to their own /dashboard, the system checks their secondary junction table: tenant_members. This creates internal company hierarchy.

Workspace Owner

The human who registered. Total control over their own bubble.

Exclusive Power: Modifies Stripe Billing, deletes workspace, assigns Admins.

Manager / Admin

Invited senior teammates.

Power: Can generate API root keys, modify the Visual Tagging Rules, and create scheduled automated reports. Cannot touch billing.

Viewer / Data Analyst

Invited junior teammates or external contractors.

Limitation: Read-Only API access. They can view graphs, real-time streams, and funnels, but cannot modify rules.

ClickHouse Data Isolation (The Absolute Wall)

When a standard `user` requests to view the Traffic Chart, the Next.js server pulls their `tenantId` from their secure AES-encrypted session cookie. It forces a mandatory WHERE tenant_id = 'their_id' clause into the ClickHouse SQL execution. It is physically impossible for Company A to accidentally view Company B's traffic volume.
